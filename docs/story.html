How this started

I'm Kat. I'm building Tanso. Two-person team. I did 32 customer discovery calls in two months and couldn't keep track of what anyone told me.

At a previous company I'd built a workflow in Airtable for this. Feature requests flowed into user stories, grouped into epics, linked to a calendar and internal feedback. It worked. But it was all manual. Tagging quotes, grouping themes, writing stories. Hours of work that I just... didn't do this time around.

Then I started playing with Karpathy's LLM Wiki. Entity pages that evolve over time. Concept pages that grow across sources. Everything traceable. And I realized: this is the same structure as my old Airtable workflow. Customer pages instead of contact records. Stories instead of feature requests. Features instead of epics. Except an LLM can maintain it.

Drop a transcript in a folder. Tell the agent to ingest it. It extracts exact quotes, tags them, writes customer pages. Then it looks across all your customers and synthesizes user stories with acceptance criteria. No database. No app. Just markdown files and a schema that tells the agent what to do.

If I delete a story the agent got wrong, it doesn't come back. If I move a story to a different feature, the agent learns the grouping. The wiki gets smarter because every edit teaches it what good looks like.

The part I didn't plan for: the wiki became the thing everyone uses. My designer opens a story and sees the actual customer words behind a feature request. My co-founder and I argue about priorities using quotes, not gut feelings. And when I use Claude Code to build, I point it at the wiki. It writes better code when it has real customer context instead of just "build me a billing page."

People who'd get value from this: founders drowning in customer call transcripts, PMs who want stories backed by evidence instead of opinions, engineers and designers who want to know why they're building something, anyone using AI to code who wants better context.
